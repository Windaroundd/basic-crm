# CLAUDE.md

Instructions for Claude Code when working in this repo. See `README.md` for
the human-facing intro.

## What this is

A clone-and-go React starter for "vibe coders". The folder structure exists
to make scaling features easy — preserve the conventions when adding code.

## Stack at a glance

Vite • React 19 • TypeScript (strict) • Tailwind CSS v4 • shadcn/ui •
TanStack Router (file-based) • TanStack Query • TanStack Table •
React Hook Form + Zod • Zustand • Supabase • Recharts • Vitest

## Folder conventions (most important)

```
src/
├── components/ui/        shadcn primitives — DO NOT edit by hand
├── components/layout/    Header, ThemeProvider, ThemeToggle, Sidebar
├── components/common/    Shared cross-feature components (Spinner, EmptyState)
├── features/<name>/      🌟 ONE FOLDER PER DOMAIN — see pattern below
├── hooks/                Cross-feature hooks (useDebounce, useMediaQuery)
├── lib/                  Third-party clients + utils (supabase, env, utils, query-client)
├── routes/               TanStack Router file-based — keep THIN
├── stores/               Zustand global stores — UI state only
├── types/                Shared types (database.ts is auto-generated)
└── test/                 Vitest setup
```

### Feature folder pattern

When adding a new feature `posts`, create:

```
src/features/posts/
├── api.ts                  Supabase calls (fetchPosts, createPost, deletePost)
├── schemas.ts              Zod schemas + inferred TS types
├── components/             Feature-only components (PostList, PostCard)
├── hooks/                  Feature-only hooks (usePosts, usePost)
├── types.ts                (Optional) Feature-only types not derived from schemas
└── index.ts                Barrel export — public API of the feature
```

See `src/features/auth/` for a complete working example.

### Routes should be thin

A route file imports a component from a feature and renders it. No business
logic in route files.

```tsx
// src/routes/posts.tsx — GOOD
import { createFileRoute } from '@tanstack/react-router'
import { PostList } from '@/features/posts'
export const Route = createFileRoute('/posts')({ component: PostList })
```

## Rules to follow

- **Features don't import from other features.** If two features share
  something, lift it to `lib/`, `hooks/`, or `components/common/`.
- **Server state goes in React Query.** Never cache API responses in Zustand
  or `useState`. Use `useQuery` / `useMutation` with `supabase` calls.
- **Client UI state goes in Zustand or `useState`.** Zustand only when state
  is shared across distant components. `useState` for local state.
- **Form state goes in React Hook Form.** Use `zodResolver` for validation.
  Don't use TanStack Form (it was removed).
- **URL state goes in TanStack Router search params.** For filters,
  pagination, tabs that should be shareable via URL.
- **Theme: use `next-themes`.** Don't roll your own — `ThemeProvider` and
  `ThemeToggle` are wired up in `components/layout/`.
- **Path alias: `@/*` → `src/*`.** Always use `@/` imports, never relative
  paths that climb directories (`../../`).

## What NOT to do

- ❌ Don't edit files in `src/components/ui/` — they're shadcn-managed. To
  update a component, re-run `npx shadcn@latest add <name> --overwrite`.
- ❌ Don't edit `src/routeTree.gen.ts` — auto-generated by the router plugin.
- ❌ Don't install `axios`, `redux`, `react-router-dom`, `react-query` (v3),
  `framer-motion`. We use `fetch`/Supabase, Zustand, TanStack Router,
  TanStack Query v5, and `motion` (only if needed) respectively.
- ❌ Don't add shadcn `form` via CLI — it's already in `src/components/ui/form.tsx`.
- ❌ Don't add `@tanstack/react-form` — the project standardized on React Hook Form.
- ❌ Don't commit `.env` (only `.env.example`).

## Code style

- Prettier is the source of truth (no semicolons, single quotes, trailing
  commas). Run `npm run format` before committing.
- TypeScript is strict — no `any` unless absolutely necessary. Prefer
  `unknown` + narrowing.
- Components: function declarations, named exports. PascalCase for
  components, kebab-case for files (`sign-in-form.tsx`).
- Zod schemas: name the schema and export the inferred type:
  ```ts
  export const postSchema = z.object({ ... })
  export type Post = z.infer<typeof postSchema>
  ```
- React Query keys: tuple style, prefixed by feature:
  ```ts
  queryKey: ['posts', 'list'] as const
  queryKey: ['posts', 'detail', id] as const
  ```

## Adding a new shadcn component

```bash
npx shadcn@latest add <component>
```

After install, if it needs a Provider (e.g. `tooltip`), wire it in
`src/main.tsx`.

## Adding a Supabase migration

```bash
# 1. Make schema changes in Studio (local) or SQL
# 2. Generate migration from the diff
npm run db:diff add_posts_table
# 3. Apply + regen types
npm run db:reset && npm run db:gen:local
```

## Testing

- Tests live next to source files (`button.test.tsx` next to `button.tsx`).
- Use Testing Library queries (`getByRole`, `getByLabelText`) — avoid
  `getByTestId` unless there's no semantic alternative.
- Mock Supabase calls in tests; don't hit the real database.

## Pre-commit hook

Husky + lint-staged runs Prettier on staged files automatically. Don't bypass
with `--no-verify` unless you have a specific reason.
